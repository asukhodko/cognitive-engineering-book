# Выжимка для книги

## Главная идея

Самопланирование - это не список дел и не календарь сам по себе. Это способ управлять конкуренцией задач за внимание, энергию, время и обязательства. Хорошая система планирования должна снижать тревожную незавершенность, делать реальный выбор видимым, ограничивать WIP, защищать глубокую работу и оставлять место для восстановления.

Короткая формула для будущего текста книги:

`capture -> clarify -> choose -> limit WIP -> execute -> review -> recover -> adjust`

## Что добавить в следующую версию

1. Развести хранение задач и выбор задач. Система может идеально хранить всё и при этом плохо помогать выбирать, что делать сейчас.
2. Ввести WIP как личную и командную когнитивную переменную: слишком много незавершенного ломает внимание, увеличивает переключения и создает ложную занятость.
3. Показать планирование как ритм: daily triage, weekly review, portfolio review, energy review, retrospective.
4. Связать персональное планирование с восстановлением: psychological detachment, защита сна, окончание рабочего дня, буферы, отказ от бесконечной доступности.
5. Для команд сделать акцент не на занятости людей, а на потоке ценности: видимость работы, зависимости, WIP limits, explicit policies, meetings hygiene, DORA/flow metrics.

## Практическая модель

Для личности:

- Capture: выгружать задачи из головы во внешнюю систему.
- Clarify: превращать туманное "надо" в следующий физический шаг и критерий готовности.
- Choose: выбирать не по громкости тревоги, а по сроку, ценности, энергии, зависимости и риску деградации сфер жизни.
- Limit WIP: держать ограниченное число активных задач, остальное парковать в backlog/waiting/someday.
- Time/energy blocks: защищать блоки для глубокой работы и учитывать ресурс дня.
- Review: раз в неделю пересматривать обязательства, сферы, проекты, долги и восстановление.
- Recover: планировать окончание работы, паузы и восстановление так же явно, как задачи.

Для команды:

- Сделать работу видимой: доска, статусы, owner, next step, blocked reasons.
- Ограничить WIP и зависшие задачи.
- Разделять срочность и важность через classes of service.
- Уменьшить meetings debt: цель, agenda, decision log, async-first там, где можно.
- Измерять не занятость, а поток: lead time, cycle time, deployment frequency, change failure rate, recovery time, blockers.
- Проводить retrospectives/postmortems без поиска виноватых, но с action items.

## Связь с другими материалами

- `M-2026-06-25-025`: распределение времени, энергии и денег дает портфельную рамку.
- `M-2026-06-19-019`: восприятие времени и результативность объясняет цену плохого temporal design.
- `M-2026-06-25-029`: ментальная выносливость показывает, почему планирование без восстановления разваливается.
- `M-2026-06-27-034`: soft/meta-skills нужны для командного планирования, договоренностей, обратной связи и управления ожиданиями.

## Предосторожности

- Не превращать раздел в обзор приложений для задач.
- Не обещать универсальную систему планирования для всех контекстов.
- Не сводить командную продуктивность к индивидуальной дисциплине.
- Не смешивать занятость с прогрессом.
- Не использовать raw `safe_urls` как готовую библиографию без проверки.
