# Связка глав 32-33

## Назначение проверки

Проверить, что переход от главы 32 к главе 33 не превращает практикум в набор советов или мотивационных историй, а действительно проверяет личный когнитивный контур на разных типах ситуаций.

## Что делает глава 32

Глава 32 собирает личный когнитивный контур:

```text
карта треков
-> активный WIP
-> вход
-> фокус-блок
-> инструменты и ИИ по роли
-> рабочий журнал
-> checkpoint
-> review
-> восстановление
```

Главный результат главы 32: читатель понимает, как встроить диагностику задачи в повторяемую систему, где важные треки видны, вход начинается с внешнего state, работа дает срез, выход оставляет checkpoint, а review и восстановление поддерживают следующий вход.

## Что должна сделать глава 33

Глава 33 должна проверить модель на разных ситуациях:

- разработчик возвращается к сложной задаче после прерывания;
- человек прокрастинирует из-за высокой цены входа;
- лид перегружен несколькими тяжелыми треками;
- сотрудник потерял мотивацию;
- человек использует ИИ как обход мышления;
- команда живет в постоянных срочных переключениях;
- человек восстанавливается после перегруза.

Каждый кейс разбирается по схеме:

```text
ситуация
-> ценность
-> угроза
-> цена усилия
-> управляемость
-> режим действия
-> первое вмешательство
-> feedback
-> граница
```

Главный результат главы 33: читатель видит, как не перепутать сигнал с причиной, выбрать первый ход, получить feedback и увидеть границу личного, командного или организационного уровня.

## Проверка учебной связки

| Переход | Проверка |
| --- | --- |
| Личный контур -> практические кейсы | Глава 33 начинает с вопроса, как применять контур и диагностику к смешанным ситуациям, а не с новой теории. |
| Карта треков -> перегруженный лид | Кейс лида проверяет WIP, invisible work, карту треков, active deep-slice и границы личного уровня. |
| Checkpoint -> разработчик после прерывания | Кейс разработчика показывает цену потерянного goal state и роль внешнего state при возвращении. |
| ИИ внутри контура -> ИИ как обход | Кейс ИИ проверяет pre-AI trace, роль ИИ, post-AI проверку, авторство решения и сохранение навыка. |
| Восстановление -> восстановление после перегруза | Кейс показывает восстановление как возвращение безопасности и управляемости, а не новый productivity-рывок. |
| Командная среда -> срочные переключения | Кейс команды переводит прерывания в формат срочности, WIP, external state и recovery debt. |
| Практикум -> границы модели | Каждый кейс имеет границы, что готовит главу 34 о том, чего модель не объясняет. |
