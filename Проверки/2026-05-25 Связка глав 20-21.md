# Проверка связки глав 20-21

Дата: `2026-05-25`.

Связка: [[../Главы/20-Продуктивность-без-самоизноса]] -> [[../Главы/21-Фокус-WIP-и-переключения]].

## Что должна сделать связка

Глава 20 вводит продуктивность как устойчивый режим:

```text
продуктивность = ценный сдвиг + сохраненный следующий вход
```

Глава 21 должна объяснить, почему следующий вход становится дорогим, если тяжелые треки одновременно живут в голове, а не во внешнем состоянии.

Переход:

```text
продуктивность сохраняет вход
-> WIP и переключения определяют цену входа
-> внешний контейнер состояния защищает от полного повторного разгона
```

## Проверка перехода

| Вопрос | Ответ |
| --- | --- |
| Есть ли явный мост из главы 20? | Да. Глава 21 начинается с формулы "ценный сдвиг + сохраненный следующий вход" и разворачивает, что именно делает вход дорогим. |
| Не повторяет ли глава 21 главы 4-6? | Нет. Главы 4-6 вводят внешний контекст, журнал и ритуалы. Глава 21 использует их как инструменты управления WIP, переключениями и прерываниями. |
| Не повторяет ли глава 20? | Нет. Глава 20 говорит о продуктивности и восстановлении режима. Глава 21 разбирает активные контексты, task switching, срезы продвижения и срочность. |
| Есть ли новая центральная схема? | Да: меньше активного WIP -> глубже вход -> срез продвижения -> checkpoint -> внешний контейнер -> дешевле возврат. |
| Есть ли контрастная схема? | Да: много активных треков -> переключения -> мелкие входы -> нет среза -> контекст холодеет -> aging -> тревога -> новые переключения. |
| Есть ли практический выход? | Да: WIP-карта внимания, правило одного среза, протокол переключения и формат срочности. |

## Что глава 21 добавляет

Новые понятия:

- WIP в задачах;
- WIP в голове;
- рабочая сфера;
- task set;
- switch cost;
- attention residue;
- resumption lag;
- минимальная полезная единица глубокого внимания;
- срез продвижения;
- внешний контейнер состояния трека;
- severity срочности;
- aging активной задачи.

Главное новое различение:

```text
портфель важных треков может быть широким,
но текущий глубокий контакт должен быть узким
```

Это защищает главу от ложного вывода "нужно заниматься только одним делом в жизни".

## Проверка последовательности

Последовательность выдержана:

```text
20: продуктивность должна сохранять будущий вход
21: будущий вход разрушается активным WIP и прерываниями
22: правила фокуса выполнимы только при достаточном ресурсном режиме
```

Так глава 21 не закрывает блок продуктивности, а подготавливает следующую главу про ресурсность, силу и ритуалы.

## Что проверить при ревизии

1. Не слишком ли длинный раздел о командном фокусе для личного учебника.
2. Достаточно ли ясно разведены "срез продвижения" и "checkpoint".
3. Не нужно ли добавить один компактный пример WIP-карты с условными задачами.
4. Не звучит ли severity срочности как жесткий корпоративный регламент вместо инженерного принципа.
5. Не стоит ли часть источников по software engineering перенести в будущую главу 30, оставив здесь только базовый уровень.
