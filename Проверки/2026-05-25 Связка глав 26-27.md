# Связка глав 26-27

Дата: `2026-05-25`.

Связка: [[../Главы/26-ИИ-как-усилитель-и-как-обход-мышления]] -> [[../Главы/27-Как-работать-с-ИИ-не-отдавая-ему-субъектность]].

## Что должна сделать связка

Глава 26 вводит развилку:

```text
ИИ как усилитель мышления
ИИ как обход мышления
```

Глава 27 должна превратить эту развилку в рабочий порядок:

```text
до ИИ
-> роль ИИ
-> проверка
-> решение человека
-> след
-> сохранение навыка
```

## Проверка перехода

| Вопрос | Ответ |
| --- | --- |
| Есть ли явный мост из главы 26? | Да. Глава 27 начинается с развилки "усилитель / обход" и объясняет, что теперь нужен практический режим. |
| Не повторяет ли глава 26? | Нет. Глава 26 объясняет рамку и риски, глава 27 дает операционный порядок работы. |
| Не стала ли глава 27 каталогом промптов? | Нет. Промпты используются как примеры ролей и проверок, но объект главы - человеческая петля вокруг ИИ. |
| Удержана ли субъектность как проверяемая функция? | Да. Субъектность разведена на цель, контекст, критерий, проверку и авторизацию решения. |
| Есть ли визуальная опора? | Да. Главная схема показывает контур `до ИИ -> роль -> диалог -> проверка -> решение -> след`. |
| Есть ли практическая таблица? | Да. Есть таблица ролей ИИ и таблица признаков потери субъектности. |
| Есть ли мост к обучению и опыту преодоления? | Да. Глава прямо удерживает полезное трение, recall, self-explanation и первую попытку. |
| Есть ли мост к разработке? | Да. Есть отдельный блок "код не равен пониманию" с режимом bugfix. |
| Есть ли переход к лидерству? | Да. Финал переводит ту же логику в проектирование среды действия. |

## Что глава 27 добавляет

Новые понятия:

- субъектность как рабочая функция;
- минимальный pre-AI trace;
- роль ИИ;
- post-AI verification;
- след решения после ИИ;
- учебный режим ИИ;
- производственный режим ИИ;
- защита полезной трудности.

Главное новое различение:

```text
субъектность не означает делать без ИИ;
субъектность означает не отдавать ИИ цель,
критерий,
проверку,
решение
и присвоение результата
```

## Что проверить при будущей ревизии

1. Не слишком ли много примеров запросов; они не должны вытеснять объяснение механизма.
2. Достаточно ли ясно разведены учебный и производственный режимы.
3. Не нужно ли добавить отдельную мини-таблицу по типам задач: low-risk, high-risk, learning, strategic.
4. Не стоит ли часть developer-specific материала перенести в будущие практические кейсы главы 33.
5. Не нужно ли перед финальной редакцией обновить источники по AI offloading и deskilling, потому что зона быстро меняется.
