# Export privacy pass 1-36 - главы и служебные слои

Дата: 2026-05-25.

Статус: закрыто для текущего состояния `ready-for-review`. Этот проход не переводит главы в `done`.

## Задача

Проверить, не переносит ли учебник в читательский слой приватную рабочую конкретику:

- внутренние названия проектов, команд, систем и материалов;
- имена, закрытые ссылки, тикеты и узнаваемые рабочие эпизоды;
- детали доступов, пароли, токены, ключи, credential hints;
- прямые локальные пути, которые не должны попадать во внешний материал;
- служебные материалы vault, если они могут быть случайно включены в экспорт.

## Область прохода

Проверены четыре слоя:

1. главы 1-36 как reader-facing слой;
2. паспорта и карты объяснения как редакционный слой;
3. источниковые пакеты и реестры как source layer;
4. верхние служебные файлы учебника: архитектура, главный файл и реестр глав.

## Результат по главам 1-36

В главах не найдено прямых внутренних названий `T-Bank`, `tbank`, `Spirit-Code`, `TEAMLEAD`, `1x1`, `DPVCS`, имен workspace, закрытых ссылок, URL, токенов, паролей или ключей доступа.

Найденные совпадения являются учебными и не требуют правки:

- `тикет`, `ревью`, `ключ` используются как обычные рабочие понятия;
- слова вроде "включенность", "переключение", "доступность" попадают в широкий scan из-за морфологии, но не являются privacy-риском;
- главы 29-30 прямо фиксируют границу: внутренние лидерские материалы используются только санитаризированно;
- глава 36 содержит защитный пункт о проверке приватности, а не приватную информацию.

Вывод: reader-facing главы 1-36 пригодны для дальнейшей редакционной работы как санитаризированный текстовый слой.

## Результат по служебным слоям

В служебных материалах найдены внутренние пути и локальные названия. Это ожидаемо: они нужны для трассировки источников и связи с vault.

К таким материалам относятся:

- верхняя архитектура и главный файл учебника;
- реестр глав;
- паспорта глав 21, 28, 29, 30 и связанные карты;
- источниковые пакеты, где внутренние материалы указаны как локальная авторская опора;
- проверки и контрольные точки.

Эти файлы не должны считаться готовым внешним материалом. Их можно оставлять внутри vault, но при публикации или передаче учебника наружу нужно либо исключить их из export set, либо вручную заменить внутренние пути и названия на обобщенные формулировки.

## Export boundary

Текущая граница экспорта:

| Слой | Можно включать во внешний учебник | Условие |
| --- | --- | --- |
| `Главы/` | Да | После финальной языковой, учебной и библиографической редакции. |
| Диаграммы внутри глав | Да | Если они не содержат внутренних названий или путей. |
| `Паспорта/` | Нет по умолчанию | Только после отдельной санитаризации. |
| `Карты объяснения/` | Нет по умолчанию | Это редакционный слой, не читательский текст. |
| `Источники/` | Нет по умолчанию | Можно использовать для финальной библиографии, но не экспортировать как есть. |
| `Проверки/` | Нет | Это audit trail работы, не часть учебника для читателя. |
| `00-06` служебные файлы | Нет по умолчанию | Для внешней версии нужен отдельный публичный front matter без внутренних ссылок. |

## Решение

Полный export/privacy pass закрывает текущий privacy-фронт.

Главы остаются в статусе `ready-for-review`, не `done`: перед переводом в `done` еще нужны сквозное чтение учебной лестницы, финальная языковая редактура и финальный библиографический список.
