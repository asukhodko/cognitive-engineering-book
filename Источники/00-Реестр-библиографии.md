# Реестр библиографии для учебника

## Назначение

Этот реестр превращает карту источников в рабочую библиографию учебника. Его задача — не хранить окончательно оформленные ссылки, а вести источники через проверку, чтение, конспектирование и привязку к главам.

Главный входной материал: [[2026-05-24 Карта источников для большого учебника]].

Полный исходный research-пакет: [[2026-05-25 Исходный research-пакет ChatGPT - карта источников]]. Его роль — сохранить сырой результат внешнего исследования; рабочие решения, статусы проверки и источниковые gates фиксируются в этой заметке и в отдельных пакетах глав.

Правила финального аппарата ссылок: [[01-Библиографический-аппарат-и-правила-ссылок]]. Этот файл задает путь `research backlog -> первоисточник -> статус -> пакет главы -> текст -> финальная библиография` и обязателен перед переводом глав из `ready-for-review` в `done`.

Статус входного материала от `2026-05-24`: принят в работу как источниковый backlog учебника. Его нельзя цитировать как научный источник, но нужно использовать при планировании чтения, подборе первоисточников и проверке полноты каждой части.

Уточнение от `2026-05-25`: research-пакет принят как управляющая карта для большого многоглавного или многотомного учебника, а не как разовая подборка литературы. При работе над каждой новой главой сначала проверять карту источников, затем этот реестр, затем отдельный пакет главы. Нельзя закрывать поздние главы по инерции черновика без проверки первоисточников и evidence policy.

## Статусы

| Статус | Значение |
| --- | --- |
| `to-verify` | Источник пришел из research-пакета и требует проверки по первоисточнику. |
| `verified` | Источник найден, авторы, год, название и место публикации проверены. |
| `read` | Источник прочитан достаточно для использования в главе. |
| `excerpted` | Выписаны ключевые тезисы, ограничения и пригодные формулировки. |
| `integrated` | Источник уже использован в тексте главы. |
| `deferred` | Источник полезен, но не нужен в ближайшем цикле написания. |

## Правила использования

- Не переносить в учебник утверждения из research-пакета как проверенный факт, пока источник не имеет статус `verified`.
- Для новых и быстро меняющихся работ по ИИ, нейронауке и software engineering явно отмечать дату проверки.
- В главе различать роль источника: фундаментальная теория, обзор, мета-анализ, эксперимент, клиническая граница, практический фреймворк, emerging concern.
- Не использовать один источник как универсальное объяснение целого раздела.
- Если источник поддерживает практическую рекомендацию, отдельно проверить уровень доказательности и условия применимости.
- Перед написанием или ревизией главы проверять, есть ли для нее отдельный источниковый пакет. Если пакета нет, сначала собрать минимум: ключевые первоисточники, роль каждого источника, статус проверки, границы доказательности.
- Перед финальной редакцией главы сверять ссылки с [[01-Библиографический-аппарат-и-правила-ссылок]]: убрать `turn...`-следы, назначить evidence label для практических выводов и отметить дату проверки для быстро меняющихся источников.

## Evidence policy

| Зона | Базовая редакционная позиция |
| --- | --- |
| Strong evidence | Можно строить главу, но без чрезмерных обещаний: self-efficacy, retrieval practice, implementation intentions, sleep/exercise, recovery from work, interruptions, cognitive offloading. |
| Context-dependent evidence | Писать с условиями применимости: mindfulness, HRV, habit under stress, AI productivity effects. |
| Mixed / weak evidence | Не превращать в рекомендации: omega-3, БАДы, широкие "нейротренинговые" обещания. |
| Fast-moving evidence | Помечать дату проверки и обновляемость: ИИ в software engineering, AI offloading, deskilling. |
| Clinical boundary | Не превращать в самостоятельные протоколы: apathy, anhedonia, depression-like states, medication topics, burnout beyond occupational framing. |
| Methodological boundary | Перед практическим выводом проверять design, population, outcome, effect size, uncertainty, risk of bias, body of evidence and bridge to practice. |
