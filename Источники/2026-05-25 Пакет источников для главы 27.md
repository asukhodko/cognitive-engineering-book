Ключевая редакционная задача: глава 26 дала рамку `усилитель / обход`. Глава 27 должна дать рабочий режим. Она не должна стать каталогом промптов, списком AI tools или обещанием "правильный запрос решит все". Ее задача - описать практику сохранения субъектности:

```text
своя постановка
-> запрос к ИИ
-> проверка
-> решение
-> след в рабочем журнале
-> сохранение навыка
```

## Что этот пакет дает главе

Глава должна заменить простую формулу:

```text
хороший промпт -> хороший ответ
```

на инженерную:

```text
собственная постановка
-> правильная роль ИИ
-> проверка
-> решение человека
-> след
-> сохранение навыка
```

Для этого нужны шесть опор:

1. Offloading полезен, если человек понимает, что вынесено наружу и как это проверяется.
2. Автоматизация требует calibrated reliance: где можно опереться, где нужно проверять, где нужно сначала подумать самому.
3. ИИ особенно опасен там, где создает связный ответ до человеческой постановки задачи.
4. Учебные задачи требуют сохранения полезного трения: recall, self-explanation, выбор проверки и присвоение результата.
5. Инженерные задачи требуют внешнего следа: контекст, гипотеза, ограничения, тест, решение, риск.
6. Чем выше цена ошибки и чем слабее возможность проверки, тем меньше ИИ должен быть финальным ответчиком и тем больше - источником вопросов, альтернатив и проверок.

## Обязательные различения

| Понятие | Что важно не перепутать |
| --- | --- |
| Субъектность | Не мистическое качество, а удержание цели, критерия, проверки, решения и ответственности. |
| Помощь | ИИ уменьшает лишнее трение, расширяет варианты или помогает проверить мысль. |
| Подмена | ИИ делает постановку, гипотезу, recall, проверку и решение вместо человека. |
| Промпт | Способ задать роль и контекст, а не магическая команда получения истины. |
| Черновик | Материал для проверки и переработки, а не решение. |
| Тренажер | ИИ задает вопросы, дает подсказки и контрпримеры, но не снимает всю учебную трудность. |
| Оппонент | ИИ атакует слабые места плана, но человек проверяет качество критики. |
| Проверка | Источник, тест, пример, контрпример, воспроизведение, объяснение своими словами. |

## Что не делать в главе

- Не превращать главу в список промптов.
- Не писать, что субъектность сохраняется одним волевым решением.
- Не советовать "сперва всегда делай без ИИ": режим зависит от задачи.
- Не советовать "пусть ИИ сделает черновик всего": для учебных и высокорисковых задач это может убрать нужную трудность.
- Не подавать ИИ как нейтральный ускоритель всех рабочих операций.
- Не превращать практические режимы в моральный тест "сам думал / не сам думал".
- Не использовать свежую AI productivity literature как вечный закон.
- Не давать рабочие советы без проверочного шага и следа в журнале.

## Минимальные тезисы для черновика

1. Субъектность в работе с ИИ - это удержание цели, критерия качества, проверки, решения и авторства результата.
2. До ИИ нужно оставить минимум собственного следа: задача, контекст, неизвестное, гипотеза, критерий полезного ответа.
3. Во время ИИ нужно задавать не только результат, но и роль инструмента: справочник, оппонент, тренажер, черновик, проверка.
4. После ИИ нужен обратный ход: проверить, объяснить своими словами, решить, что принято, и зафиксировать след.
5. Для обучения ИИ должен чаще давать вопросы и подсказки, чем готовые ответы.
6. Для разработки ИИ должен входить в контур тестов, локального контекста и сопровождения, а не только генерации кода.
