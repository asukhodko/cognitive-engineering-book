# Пакет источников для главы 21

## Статус

Дата проверки: `2026-05-25`.

Глава: [[../Главы/21-Фокус-WIP-и-переключения]].

Назначение: источниковая опора для главы о фокусе, WIP, переключениях, прерываниях и повторном входе в задачу. Глава продолжает вывод главы 20: если продуктивность сохраняет следующий вход, то нужно отдельно разобрать, что делает следующий вход дорогим.

Ключевая редакционная задача: не писать главу как набор советов "не отвлекайся". Нужно показать механизм:

```text
много активного WIP
-> частые переключения
-> каждый вход мелкий
-> нет среза продвижения
-> контекст холодеет
-> задачи стареют
-> тревога и срочность растут
-> еще больше переключений
```

Рабочая альтернатива:

```text
меньше активного WIP
-> глубже вход в один трек
-> видимый срез продвижения
-> checkpoint на выходе
-> внешний контейнер состояния
-> дешевле возврат к отложенным трекам
```

## Что этот пакет дает главе

Глава должна заменить простую формулу:

```text
чтобы успевать больше, нужно лучше концентрироваться
```

на инженерную:

```text
чтобы фокус работал,
нужно управлять количеством активных контекстов,
ценой переключения,
качеством выхода
и правилами прерывания
```

Для этого нужны пять опор:

1. Рабочая память и cognitive control ограничены: несколько тяжелых task sets не могут честно быть полностью активными одновременно.
2. Переключение стоит не только минут, но и восстановления цели, правил, контекста и внимания.
3. Внешний контейнер состояния превращает отложенный трек из "держать в голове" в "вернуться по cues".
4. Cognitive load in software development is a measurement problem: code metrics, familiarity, comprehension and physiological proxies do not collapse into one simple number.
5. Командная среда должна отделять настоящую срочность от шума, иначе люди вынуждены проверять каждое входящее как потенциальное переключение.

## Обязательные различения

| Понятие | Что важно не перепутать |
| --- | --- |
| WIP в задачах | Сколько единиц работы формально находится в процессе. |
| WIP в голове | Сколько тяжелых контекстов человек пытается удерживать как активные. |
| "В работе" | Статус потока; не доказательство реального ежедневного контакта внимания. |
| Глубокий фокус | Достаточно длинный контакт с одним треком, чтобы изменилось состояние задачи. |
| Срез продвижения | Значимое улучшение состояния трека; не обязательно завершение всей задачи. |
| Checkpoint | След, который делает следующий вход дешевле. |
| Переключение | Запланированная смена трека после выхода и checkpoint. |
| Прерывание | Вынужденный разрыв текущего блока до нормального выхода. |
| Срочность | Требование быстрого переключения с понятным impact, deadline и нужным действием. |
| Шум | Входящее без ясной цены ожидания, которое заставляет проверять себя за счет фокуса. |

## Что не делать в главе

- Не обещать универсальный числовой WIP-лимит.
- Не говорить, что все переключения вредны: часть ролей живет в координации, on-call и поддержке.
- Не превращать главу в moralizing о "слабой концентрации".
- Не переносить приватные рабочие детали из TEAMLEAD/T-Bank материалов.
- Не использовать популярную формулу "после любого прерывания нужно 23 минуты", как если бы она была универсальным законом.
- Не делать вывод, что хорошая команда просто "не отвлекает" людей: срочные сигналы и координация нужны, но должны иметь форму.
- Не смешивать внешний контейнер состояния с обычным TODO-списком.

## Минимальные тезисы для черновика

1. Фокус - это не героическое состояние, а управляемый режим контакта с одним рабочим контекстом.
2. WIP в голове опаснее WIP на доске, потому что он потребляет goal memory, attention и способность войти глубоко.
3. Переключение требует восстановления task set; подготовка и cues снижают цену, но не отменяют ее.
4. Тяжелый трек должен жить во внешнем контейнере состояния, а не в тревожном фоне.
5. Когнитивная нагрузка разработчика зависит не только от размера кода, но и от comprehension, гипотез, локальности контекста, uncertainty and measurement limits.
6. Минимальная полезная единица глубокого внимания заканчивается срезом продвижения и checkpoint.
7. Срочность должна быть видна до чтения длинного сообщения: канал, severity, impact, что нужно от адресата.
8. Командный фокус строится не лозунгом "не отвлекать", а правилами WIP, ожиданий, aging и формата прерываний.
