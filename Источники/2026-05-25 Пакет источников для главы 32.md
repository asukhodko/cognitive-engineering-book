# Пакет источников для главы 32

## Статус

Дата проверки: `2026-05-25`.

Глава: [[../Главы/32-Проектирование-личного-когнитивного-контура]].

Назначение: источниковая опора для главы, которая переводит разовую диагностику задачи в повторяемую личную рабочую систему. Глава 31 дала карту отдельной задачи. Глава 32 должна показать, как собрать устойчивый контур, в котором карта треков, рабочий журнал, ритуалы входа и выхода, WIP, ИИ, feedback, review и восстановление работают вместе.

Ключевая редакционная задача: не написать еще один список productivity-советов. Глава должна собрать уже введенные элементы в систему:

```text
карта треков
-> выбор активного WIP
-> вход в задачу
-> фокус-блок
-> рабочий журнал
-> инструменты и ИИ по правилам
-> checkpoint и выход
-> review
-> восстановление
-> следующий вход
```

Глава должна показать не "идеальную систему жизни", а минимальный контур, который снижает цену возвращения к важной работе и вовремя показывает, где система начала ломаться.

## Что этот пакет дает главе

Глава должна заменить простую формулу:

```text
нужна хорошая система продуктивности
```

на инженерную:

```text
личный когнитивный контур =
    внешняя память
    + правила входа
    + ограничение активного WIP
    + рабочий журнал
    + проверяемые срезы
    + правила ИИ и инструментов
    + checkpoint
    + review
    + восстановление
```

Для этого нужны семь опор:

1. Контур должен снижать цену следующего входа, иначе это архив или трекер, а не рабочая система.
2. Активный WIP нужно выбирать явно: нельзя держать все важное в голове как одинаково активное.
3. Вход должен начинаться не с настроения, а с внешнего состояния трека и первого среза.
4. Выход должен оставлять checkpoint, иначе следующий вход снова платит полную цену восстановления.
5. ИИ должен входить в контур по роли и с проверкой, а не становиться первым автором постановки.
6. Review нужен для настройки контура, а не для самообвинения или красивой отчетности.
7. Восстановление является частью контура; если оно постоянно вытесняется, система работает за счет долга.

## Обязательные различения

| Понятие | Что важно не перепутать |
| --- | --- |
| Личный когнитивный контур | Повторяемая рабочая система входа, действия, выхода, следа, review и восстановления. |
| Productivity-система | Может быть полезной, но часто измеряет задачи и время; контур измеряет доступность действия и цену следующего входа. |
| Инструмент | Приложение, шаблон, ИИ, календарь или заметка; сам по себе не является контуром. |
| Карта треков | Внешняя карта активных, ожидающих и фоновых рабочих контекстов. |
| Активный WIP | То, что прямо сейчас получает глубокий контакт и срез продвижения. |
| Review | Настройка системы по сигналам, а не суд над человеком. |
| Восстановление | Условие будущего входа, а не награда после выполнения плана. |
| Минимальный контур | Небольшая повторяемая связка, которую реально использовать в плохой день. |

## Что не делать в главе

- Не превращать главу в список приложений, шаблонов, утренних ритуалов и трекеров.
- Не повторять главы 4-6 как инструкцию по журналу.
- Не повторять главу 21 как полный текст про WIP и переключения.
- Не повторять главы 26-27 как руководство по ИИ.
- Не обещать, что личный контур решит хроническую перегрузку, депрессию, тревогу, болезнь или организационный конфликт.
- Не делать из review практику самообвинения.
- Не строить систему, которую можно поддерживать только в хорошем состоянии.
- Не переносить личные productivity-заметки как рецепты без перевода в общие принципы.

## Минимальные тезисы для черновика

1. Личный когнитивный контур нужен, чтобы полезные элементы учебника работали вместе, а не оставались отдельными приемами.
2. Контур начинается с карты треков: что активно, что ждет, что заблокировано, что хранится в фоне и что требует восстановления.
3. Вход в работу должен поднимать внешний state, а не заново собирать задачу из памяти.
4. Активный WIP должен регулярно получать срез продвижения; иначе задача стареет и тревожит.
5. Выход из блока должен оставлять checkpoint, который делает следующий вход дешевле.
6. ИИ и инструменты должны работать внутри человеческого контура: цель, роль, проверка, решение, след.
7. Review показывает, где контур перестал работать: слишком много WIP, нет срезов, нет recovery, нет следа после ИИ, review стал обвинением.
8. Минимальный контур лучше большой системы, которая ломается в первый тяжелый день.
